# Issue

When using sparql-parser, updates including triples with large string objects fail.
The built-in behaviour of Ember Data is to send all fields when saving an object (presumably to handle concurrent edits).
This means that any update to a field of such an object will trigger the issue.

## Reproduction

The issue comes from how sparql-parser [handles updates](https://github.com/mu-semtech/sparql-parser/blob/040b90708c5d8d21eac2a5dee441d52196133ba2/updates/handle-update-unit.lisp#L149), specifically, how it checks for values being the same after updates.
For this check it uses a select with a values clause, which seems to have a character limit of 8158 characters per 'row'. Strangely, the data needs to match to trigger the issue.

## Possible solutions

### Enable the sparql-parser string-max-size feature

This transparently handles large strings as files instead of storing them in the triplestore.
The stability of this as well as the impact on debugging needs to be investigated.

### Making large string fields immutable

This is already the case for most of the fields in question as a new editor document is created for each save, to allow for reverting to previous versions.
We would need to modify the behaviour of Ember Data / Warp Drive to no longer send these fields on update.
This is potentially dangerous as it is easy to miss instances where updates need to happen.

### Handling large strings as files in GN

This handles the problem explicitly and means that content is in a consistent place for debugging.
Care must be taken about integrations between services and applications as there are other stacks which make use of GN data.
